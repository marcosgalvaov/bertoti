1 - SOFTWARE É INTANGÍVEL

"Programação" e "Engenharia de Software" possuem ênfases e aplicações diferentes.
Engenharia usa o conhecimento teórico aplicado para criar algo real. Na engenharia de software é a mesma coisa, apesar de o que é criado ser intangível. 
Engenharia de software é mais nova de todas.
A prática e teoria da engenharia de software não é nem perto de ser tão rigorosa. Comparando com outros tipos de engenharia mais "sérias".
Porém, como os softwares estão cada vez mais presentes na nossa vida, é preciso começar a tomar medidas e métodos mais rigorosos.
Engenharia de software não é apenas programar, mas é também todas as ferramentas e processos que uma organização usa para criar e manter o código sempre.


2 - ENGENHARIA DE SOFTWARE = PROGRAMAÇÃO + TEMPO, ESCALABILIDADE E TRADE-OFFS

Programação integrada ao longo do tempo.

Fundamentos para o design, arquitetura e escrita do código:

1 - Time and change: Como o código vai precisar se adaptar ao longo da vida dele

2 - Scale and growth: Como uma organização precisará se adaptar à medida que evoluí.

3 - Trade-offs and costs: Como a organização faz decisões, baseado nos fundamentos anteriores.

Engenharia da programação da década de 80/90 pararam de funcionar após o lançamento do primeiro iphone. Isso porque a requisição passou a ser muito grande. Se tornou de maior acesso.
O software não para, ele se modifica pela quantidade de requisições.

3 - REQUISITOS NÃO FUNCIONAIS

Escalabilidade - Suporte para muitas conexões ao mesmo tempo. Aguentar o número de usuários utilizando o sistema. Kafka criado pelo LinkedIn em 2009 - Ferramenta pra escalar sistemas (usado pela maioria dos bancos).

Testabilidade - Os testes feitos durante o desenvolvimento do sistema, o quão completo esses testes são e o quão bem eles rodaram durante o desenvolvimento. Testar partes individuais e ver como elas funcionam.

Modularidade - Como o sistema é organizado e montado. Separando ele em componentes e partes separadas, cada módulo tem uma funcionalidade específica e funciona de forma independente.

Safety - O sistema precisa ser seguro, a prova de erros, principalmente se erros afetarem vidas. Como o caso de sistemas médicos, aplicações em aviões, etc.

Robustez - A habilidade do sistema em continuar funcionando apesar de erros. Se o software continua performando bem, sem cair e sem perder performance.


4 - TRADE-OFFS

1) Minecraft Java x Minecraft Bedrock
- Bedrock apresenta maior portabilidade e escalabilidade.
-  Java apresenta maior robustez e consistência.

Enquanto a versão Bedrock roda em diversos tipos de dispositivos (computadores, dispositivos móveis, videogames), principalmente em dispositivos mais leves, a versão Java roda apenas em computadores com sistema windows, IOS, linux e chromeOS. Apesar disso, a versão Java apresenta uma compreensão maior em segurar erros e consistência com o seu sistema, enquanto a versão Bedrock apresenta mais erros.


2) Android x IOS
- Android apresenta maior flexibilidade, portabilidade, usabilidade 
- IOS apresenta maior performance, segurança, robustez.
